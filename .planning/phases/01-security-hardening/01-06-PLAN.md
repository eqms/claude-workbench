<objective>
Phase B of the two-phase signing rollout: wire `.verifying_keys([RELEASE_VERIFYING_KEY])` into
both update functions in `install.rs`. The public key is loaded at compile time via `include_bytes!`.

**DO NOT START THIS PLAN until at least 2 signed releases have been published by Plan 05.**

Purpose: With CI now signing every release archive (Plan 05), the client can require signatures.
If this plan is merged before signed archives exist, every existing user's auto-update will fail
with `Err(NoSignatures(...))`. The temporal gap between Plan 05 and Plan 06 is the security
control — it is mandatory, not optional.

Output: Both `perform_update_sync()` and `perform_update_to_version_sync()` call
`.verifying_keys([RELEASE_VERIFYING_KEY])`. The constant is a `[u8; 32]` loaded from
`signing/claude-workbench-pub.bin` via `include_bytes!`. The existing error handling in
both functions propagates `Err(Signature(...))` and `Err(NoSignatures(...))` through the
existing `UpdateResult::Error(msg)` arm — no structural changes needed.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Release download → binary self-replace | Compromised or tampered archive must be rejected before self-replace |
| compile-time pub.bin → runtime verification | Key must match what CI used to sign; rotation requires recompile |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | Compromised GitHub release asset | mitigate | `.verifying_keys()` in both update functions — wrong signature returns `Err(Signature)` |
| T-06-02 | Tampering | Old unsigned release downgrade | mitigate | `Err(NoSignatures)` returned for archives lacking signature comment; update aborted |
| T-06-03 | Tampering | Public key substitution at runtime | mitigate | `include_bytes!` bakes key into binary at compile time — no runtime fetch, no PKI |
</threat_model>
